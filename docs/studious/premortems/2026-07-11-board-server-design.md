# Pre-mortem — local stdlib SSE board server

| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | The SSE poll reads lines appended to `events.jsonl` since the last tick and parses each by hand, but `append_event()` writes via non-atomic `>> "$file"` (`bin/gate-ledger:188`). A reader polling mid-append can read a truncated trailing line and crash or silently drop that transition. | Force a mid-write read in a test (append a long line in chunks while polling); confirm the reader skips an unparseable trailing line and re-reads it next tick rather than raising or dropping it. |
| 2 | technical | The feed is described as "chronological," but `reference/events-format.md` warns physical line order across concurrent writers may not match wall-clock `at` order. A reader that emits events in file-read order shows an out-of-order feed when two stories transition near-simultaneously. | Inspect the schema-builder: it must sort the event feed by the `at` field, not by file line order. Reproduce by writing two events with the later `at` physically first. |
| 3 | technical | "Delta within one tick" depends on the poll detecting every change. If change-detection keys on blackboard `mtime` alone, a content change that lands within the same coarse mtime granularity (or an `events.jsonl` append that leaves the blackboard mtime unchanged) is missed and the transition is never pushed. | Attach an SSE client during a real `/work-through` run, force a `gate-ledger record`/`epic-story-set`, and confirm a fresh snapshot arrives within one tick. Verify detection reads content/append-offset, not mtime alone. |
| 4 | product | The design satisfies "pushes an SSE delta" by pushing a fresh whole snapshot, not a computed JSON-patch. If `/gate-acceptance` or `board-ui` reads "delta" literally as an incremental patch, the acceptance criterion is disputed at merge. | At `/gate-acceptance` (product lane), confirm the "SSE delta within one tick" criterion is judged met by snapshot-on-change; check the acceptance reviewer does not expect a JSON-patch payload. |
| 5 | technical | Landing an HTTP+SSE server makes `DESIGN.md`'s "single surface / no HTTP API" statement false (design Open questions; epic pre-mortem item 5). If `DESIGN.md` is not updated, a later `/gate-audit` keeps auto-skipping the web-surface lanes (UX / frontend / a11y) that now have a real surface to audit. | Run `/gate-audit` on a board-touching changeset; if it reports the frontend/UX lanes auto-skipped while an HTTP surface exists, and `DESIGN.md` still reads "single surface," the drift materialized. |
| 6 | technical | The core safety property is that no write/mutate endpoint exists ("Read-only by construction"). A build that adds any POST handler, or reuses viva's request handler carrying its verdict-ledger write path, breaches "Propose, don't apply" at this layer — a second, web-shaped mutation path into ledger state `gate-ledger` owns exclusively. | Inspect the request handler: it must dispatch only the two documented GETs (state fetch, SSE stream) and reject every other verb; confirm no viva write-capable handler code was copied in. |
| 7 | technical | The server binds a free loopback port and prints its URL, with no lifecycle tie to the driver (out of scope by design). A stale instance from a prior epic run can keep serving an old slug's files, or a second launch collides on state, with no signal to the operator. | Launch twice (same and different epic slug); confirm each instance is scoped to exactly its slug's two files and that a killed driver does not orphan a bound port that keeps serving stale state unnoticed. |
