# Pre-mortem — Story-scoped evidence-capture hook

Story-local failure modes only — cross-story risks (Exhibit-format drift, dogfood
item zero, the shared gate-ledger choke point) live in the epic register
(`worker-evidence-and-board-epic.md`). Verified at build by `/gate-audit` (technical
lane) and at merge by `/gate-acceptance` (product lane).

| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | The exact `tool_response` field for Bash exit status is unresolved (Open questions); if the hook reads the wrong field or treats an absent field as 0, every record maps to `result: PASSED` — including genuinely failed checks — silently defeating the capturer-≠-claimant value the story exists for. | Build-time fixture test that runs a deliberately-failing verification command (e.g. a failing `pytest`) and asserts the record's `exitCode` is non-zero and `predicate.result` is `FAILED`. In the diff, confirm `hooks/evidence-capture.sh` reads the field name the build-time spike verified, not a guessed one. |
| 2 | technical | Dispatched-worker capture never fires from inside a Task subagent. The docs confirm `agent_id` is present in subagent PostToolUse input, but studious's own hook — running in the worker's process — may fail to resolve `repo_root()`/arming against the main tree's shared `.studious/`, so the subagent path captures nothing while the docs citation reads as proof it works. | A build-phase test that dispatches a real Task subagent running a verification command and asserts a record with `origin: "subagent"` lands in the shared `.studious/evidence/<slug>.jsonl` — not a docs citation. Check the test exercises the real dispatch path, per the acceptance criterion's "captured the same way, or documents why not." |
| 3 | technical | Allow-list false-negative → silent evidence gaps. The verification pattern list is conservative and generic; a consuming project's real test/check runner (studyengine's, per Open questions) isn't matched, so real verification runs leave no record and a downstream gate cites an incomplete trail with no signal that it is incomplete. | Run against the first dogfood target; diff the verification commands actually run in a story against the records captured. A verification command in the transcript with no matching record is the signature. Confirm the pattern list lives as data in the hook script (iterable), not baked into prose. |
| 4 | technical | Hot-path degradation. The hook registers against every `Bash` call in every session of every installing project, not just gated stories; a slow/hung `jq`, an unhandled error, or an emitted non-`allow` decision would add perceptible latency or break unrelated sessions. | Confirm `hooks/evidence-capture.sh` mirrors `gate-reminder.sh`'s degrade-silently posture: `have jq`/`have git` guard exits 0, no `deny`/`ask` ever emitted, no network call, any internal error path exits 0. Verify near-zero overhead on an unarmed branch. |
| 5 | technical | `outputDigest` taken over unstable output. If the digest covers raw output containing timestamps, temp paths, or PIDs, the "two runs produced identical output" property the doc claims never holds in practice and the digest becomes decorative — while a secret in the output is still hashed, not excluded. | Check which stream(s) the digest covers and whether wall-clock/path-tainted output is normalized before hashing. Lower confidence — a raw digest may be acceptable for v0 if the doc's claim is scoped to byte-identical reruns only. |
| 6 | technical | The `reference/` serialization doc drifts from what the hook actually writes. The acceptance criterion requires the byte-exact shape written to `reference/` so drift from winnow is a visible diff — but if that file is prose restating the shape rather than an artifact tested against, the hook's real output can diverge from the documented shape with no failing signal. | Check whether a test asserts a captured record validates against the shape pinned in the new `reference/` file, or whether the reference is decorative prose only. The epic register's item 1 signal (schema doc must cite the amendment's "early footprint" section) also applies. |
| 7 | product | PRODUCT.md's parked-item line is silently never edited. The design correctly defers moving "A worker-layer skill set (evidence capture and handback first) is parked…" out of "What we're NOT building" to a human post-land (propose-don't-apply) — but with no tracked owner the line stays contradicting shipped reality, confusing the next reader of PRODUCT.md. | After the epic lands, grep PRODUCT.md's "What we're NOT building" for the evidence-capture parked line; still present with the feature shipped = the follow-up was forgotten. The doc flags this as an open question — this item closes the loop at acceptance. |
