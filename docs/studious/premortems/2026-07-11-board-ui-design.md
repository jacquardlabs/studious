| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | The resolution command decides whether to include `--reset-retry <gate>` by parsing `parkPrompt`'s free-form `reason` prefix (`"<gate>: <verdict> — ..."`). A retry-cap park whose reason doesn't match that shape drops the flag, so the operator pastes a command that resumes the story with `retries[gate]` still at the cap — it re-parks on the next dispatch. The wrong copy-paste is exactly the trust-eroding surface the design warns against. | Force a retry-cap park with a hand-edited reason string and confirm the copied command still carries `--reset-retry`. Prefer keying inclusion off the structured `retries[gate] >= MAX_FIX_CYCLES` field, using the reason prefix only as corroboration. |
| 2 | technical | Fresh-eyes labeling flags any `gate-verdict` preceded in the `at`-sorted feed by a `story` event with `bumpRetryGate == <gate>`. `events-format.md` sorts at second resolution with a stable tie-break on read order; under concurrent writers on the shared epic `events.jsonl`, a bump and its re-audit verdict sharing one `at` second could sort against causal order and mislabel the re-run (fresh shown as not-fresh or vice-versa). | In the drawer's verdict trail, check a re-audit entry immediately following a same-second retry bump for a correct fresh-eyes annotation. Verify the render function's tie-break preserves causal (bump-before-verdict) order. |
| 3 | technical | Position-is-identity depends on `stories` JSON object key order surviving jq → Python → JSON → JS unchanged, and on `cmd_epic_story_set` updating a story in place rather than re-inserting it. If any layer reorders keys or re-inserts, an existing gauge moves — the exact regression the settled comment forbids ("instruments never move"). | Add a story mid-run and kill-and-resume `board-server`; confirm no existing gauge changes grid position. Inspect that the page pins position from the first snapshot's key order and never re-derives it from a later snapshot. |
| 4 | technical | "Blocked names its blocker" derives blocked from any `deps` entry whose `status != landed`. A `dropped` dependency (deliberately abandoned, never landing) renders the dependent as perpetually "OFF — ON \<dropped-dep\>", reading as a live wait rather than a dead branch — an operator waits on progress that will never come. Design Open-question #2 flags `dropped` treatment but doesn't resolve blocked-on-dropped. | Drop a story another depends on; confirm the dependent's gauge does not imply it is still waiting on upstream progress. Resolve the `dropped`-dep case in the blocked-derivation logic at build time. |
| 5 | technical | The fix-budget wedge denominator (`2`) is a page-level constant duplicating `workflows/epic-driver.js`'s `MAX_FIX_CYCLES`, not carried in the schema (design Open-question #1). If the driver's cap changes, every wedge fills to the wrong fraction silently, with no test catching the divergence. | Grep the page for the literal `2` / `MAX_FIX_CYCLES` sync comment; confirm it matches the driver constant, or that Open-question #1 was resolved by adding a `maxFixCycles` schema field. |
| 6 | technical | `DESIGN.md`'s "Surfaces" section still reads "no web UI... HTTP API" (verified at line 14). This story ships exactly that; the design correctly defers the edit to the human (Propose-don't-apply, Open-question #3). But if the epic lands with no re-extraction note or follow-up issue, a later unrelated `/gate-audit` silently starts running the web a11y/frontend/UX lanes — epic pre-mortem item 5's exact realization. | At epic finale, confirm a DESIGN.md re-extraction note or follow-up issue exists. If a subsequent unrelated PR's `/gate-audit` changes lane composition unpredicted, the drift materialized. |
| 7 | product | The board's core job — observe a live `/work-through` epic — is grounded in PRODUCT.md's product capability and "entrypoints per scope" principle, but `/work-through` is still absent from PRODUCT.md's numbered "Critical user journeys" (Initialize / Per-feature gate flow / Per-project health loop). A reviewer reading that list literally sees the board serving an undocumented flow. Third sibling design doc in this epic to flag it unaddressed. | At `/gate-acceptance` (product lane), confirm the board's job traces to PRODUCT.md; check whether "Critical user journeys" was updated to name `/work-through` or the gap was carried forward again. |
| 8 | product | CAS green-tier ("everything else worth surfacing from the event feed") is left to build-time judgment. Over-include advisory events and the load-bearing amber cautions scroll under green noise despite the severity-major sort; under-include and the board looks dead during long green stretches, undermining "at-a-glance legibility." | During a real run: with no parks, confirm the green tier shows meaningful recent activity; when the first park arrives, confirm its amber caution renders above every green entry regardless of recency. |
