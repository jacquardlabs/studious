| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | The persisted `blockingLanes` list is serialized in one lane-identifier vocabulary on one surface and compared in another on the other — `epic-driver.js` names lanes as `studious:security-auditor` (the `AUDITORS` array), while `gate-audit.md` refers to them as `@agent-security-auditor` / numbered lanes. If a `blockingLanes` list written by one surface is read by the other, condition 3 ("all entries are lanes that still exist in the current roster") either false-negatives (fails closed — safe) or the two surfaces disagree on which lanes to re-dispatch — the exact cross-surface drift acceptance criterion 5 forbids. | At merge, grep both `epic-driver.js` and `commands/gate-audit.md` for how a blocking lane is written into and read back from the ledger record. They must canonicalize lane identity to one shared spelling. A test that writes a `blockingLanes` record on one surface and asserts the other narrows to the same lane set is the proof; two independent, unshared identifier maps are the tell. |
| 2 | technical | `auditFanIn`'s in-memory carry-forward path uses the richer prior-round result it happens to hold (full Important/Track findings) instead of projecting down to the PASS-status line, so the in-run epic surface emits fuller carry-forward text than a resumed run or the standalone `/gate-audit` can ever reconstruct from the ledger's `blockingLanes`-only shape — the content drift the design's "carry-forward content parity is deliberate" paragraph exists to prevent. | Read `auditFanIn`'s carry-forward branch: it must extract only the PASS-status line ("`<lane>`: carried forward, no Confirmed Critical as of `<sha>`") and discard any Important/Track findings present in the in-memory result. A carry-forward that interpolates prior findings from the in-memory JS variable rather than from the ledger-provable fact is the finding. A test that runs one narrowed round through both surfaces and diffs the carry-forward text confirms parity. |
| 3 | technical | A carried-forward (by-design skipped) lane is fed to `joinReports` as an absent/`null` report slot instead of an explicit PASS-status entry, so line 190's `AGENT DIED — no report; this lane is UNAUDITED` path fires and floors the verdict to `NEEDS DISCUSSION` — or, in the opposite and worse direction, a lane whose agent genuinely died this round is swept into the "carried forward, clean" bucket, silently upgrading an unknown gap to an unearned PASS. | Read how the narrowed round assembles the `reports` array before `joinReports`. A carried-forward lane must arrive as its own labeled PASS-status outcome, never as a missing slot; a real death this round must still reach the existing UNAUDITED path. A test with one carried-forward lane and one died lane in the same narrowed round, asserting they receive different labels and the died one still forces the `NEEDS DISCUSSION` floor, is the tell. |
| 4 | technical | The fix-delta cross-lane pass is scoped to the whole changeset (`git merge-base HEAD origin/main`...`HEAD`) instead of the inter-round delta (prior round's recorded `sha`...`HEAD`) — a copy-paste of the auditors' scope. It then costs as much as a full lane, erasing the entire "cheap net over a small, known-risky diff" rationale, and re-flags whole-changeset findings the previously-blocking lanes already own. | Read the fix-delta pass's dispatch: its diff base must be the ledger's prior-round `sha`, not the merge-base with the default branch. A pass diffing merge-base-to-`HEAD` (the auditors' scope) rather than prior-sha-to-`HEAD` (the fix commit) is the finding. |
| 5 | technical | Condition 2's ancestor check ("recorded sha is an ancestor of current `HEAD`") is implemented as object-existence or string-equality rather than true reachability, so after a rebase/force-push/squash that leaves the old commit object dangling in the local store (reflog, another ref) the check false-passes, and the round narrows against a fix delta computed from a sha no longer on the branch — defeating the story's fail-closed guarantee (acceptance criterion 4) precisely on the rewrite case it was built to catch. | Read the ancestor test: it must be `git merge-base --is-ancestor <recorded-sha> HEAD` (exit-0 semantics), not `git cat-file -e <sha>` or a string compare of two shas. A reachability check that only proves the commit object exists, not that it is reachable from `HEAD`, is the finding. A rebase-between-rounds acceptance case must observe the full 9-lane path fire and say why. |
| 6 | product | Important/Track findings that a carried-forward (non-blocking) lane raised in the prior round silently vanish from the narrowed re-audit report — the user sees only "carried forward, no Confirmed Critical," with no reminder those non-Critical findings still stand. A persona in the retry loop who saw a round-1 Important finding may merge on the round-2 PASS having forgotten an open item a full re-run would have re-surfaced. | At acceptance, run a FIX AND RE-AUDIT cycle where a carried-forward lane raised an Important finding in round 1, then read round 2's Summary. It must remind the user that carried-forward lanes' prior-round Important/Track findings still stand (or point back to the round-1 report), not merely list the lane as PASS-status. A round-2 report that omits any trace of the prior Important finding is the tell. |
| 7 | product | The narrowed PASS reads as under-audited: a persona who sees "2 of 9 lanes re-dispatched" on a PASS distrusts it ("did it actually re-check security?"), the opposite of the confidence the gate exists to confer — most acute the first time they hit the mechanism after upgrading, with no prior mental model of why fewer lanes ran. | At acceptance, read the exact Summary a narrowed PASS emits to a first-time user. It must explain why fewer lanes ran and make clear the fix-delta pass covered the carried-forward lanes' subject matter against the fix commit — legibly enough that the persona trusts the PASS without external documentation. A Summary that states the narrowed count without justifying the coverage is the finding. |
