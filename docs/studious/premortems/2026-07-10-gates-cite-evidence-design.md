# Pre-mortem — Gates cite the branch's evidence log

- Design doc: docs/superpowers/specs/2026-07-10-gates-cite-evidence-design.md
- Branch: epic/worker-evidence-and-board--gates-cite-evidence
- SHA: 07f3151
- Date: 2026-07-10

Story-local failure modes only — cross-story risks (the shared gate-ledger
write/read choke point across this story and `evidence-capture-hook`) live in the
epic register (`worker-evidence-and-board-epic.md`, item 4). Verified at build by
`/gate-audit` (technical lane) and at merge by `/gate-acceptance` (product lane).

| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | `evidence-list` resolves the wrong branch's log under a worktree — it relies on `branch_slug()`/`repo_root()`, but a gate run inside a story worktree (as gate runs in this epic do) can anchor to the main tree's `.studious/evidence/` or a mismatched slug, so the log cited doesn't match the branch under audit. | In `bin/gate-ledger`, confirm the new `cmd_evidence_list` reuses the *identical* `repo_root()`/`branch_slug()` the write path (`cmd_evidence_append`) uses. A build-phase test that appends a record on a worktree branch and reads it back via `evidence-list` from that same worktree. |
| 2 | technical | Non-empty-but-noise output triggers the stamp — if `evidence-list` ever writes a `jq`/`git` warning or trailing whitespace to **stdout** on a logless branch, the "Evidence log for this branch" block gets stamped anyway and the "no log → byte-identical prompt" guarantee (the third acceptance clause) silently breaks. | Confirm `evidence-list` writes **zero bytes** to stdout when the file is absent or `jq`/`git` is unavailable — warnings go to stderr. Check how the commands test "empty" (`[ -s file ]` / `[ -n "$out" ]`, not `test -e`). A test asserting an empty stdout on the no-log path. |
| 3 | technical | Citation drops the contradicting-diff guard — premortem-auditor's rule resolves `predicate.result: "PASSED"` → NOT REALIZED only "with no contradicting diff evidence." If the implementer drops that guard, a stale PASSED entry (command ran early, code regressed after) cites the log to green-light a failure mode that in fact realized. | In `agents/premortem-auditor.md`, confirm the added rule keeps `FAILED` **or** "the diff otherwise showing the failure mode → REALIZED." The citation must be **additive** to the existing diff check, never a replacement for it. |
| 4 | technical | The verb ships but neither command is wired to call it — four files change independently (verb + two commands + two agents). A partial landing where the agents gain the three-way rule but the commands never run `evidence-list` leaves the block never produced and the new agent prose dead. | `grep` `commands/gate-audit.md` and `commands/gate-acceptance.md` for a literal `gate-ledger evidence-list` invocation stamped into test-auditor's / premortem-auditor's dispatch. Absence = dead wiring, and reports read exactly as pre-story. |
| 5 | product | "Attested" becomes wallpaper on every armed branch — `evidence-capture-hook`'s allow-list is conservative/incomplete, so on an armed branch many execution claims still aren't captured and each reads "attested." The flag stops meaning "notable gap" — the exact dilution the third acceptance clause prevents, reappearing one level up (armed branches instead of all branches). | On a dogfood gate run against an armed branch, count "attested" vs cited lines. A majority-attested report means the signal diluted; feeds the epic kill rule's "receipts never changed a verdict" check. |
| 6 | product | `/gate-acceptance` cites nothing when no pre-mortem register exists — acceptance routes evidence citation solely through premortem-auditor, which dispatches only when a register exists. A branch with a rich evidence log but no register (design-review skipped or non-PROCEED) gets zero citation from acceptance, though the acceptance criterion reads "for a branch that has a log." | Run `/gate-acceptance` on a logged branch with **no** premortem register. Confirm the absent citation is intended-and-communicated (no execution claim was made to back), not the feature silently not working from the user's view. |
| 7 | product | An over-eager citation misleads worse than a disclaimer — a consulting agent cites a superficially-similar entry (e.g. a narrow `pytest -k smoke` record) to back a broad "the suite passes," producing a false "verified per evidence log" that reads as stronger proof than the honest disclaimer it replaced. | In a dogfood run, spot-check each cited entry's actual `command` against the claim it backs. A citation whose captured command is narrower than the claim it supports is a mis-cite — a user-trust regression, not a win. |
