# Pre-mortem — handback skill: evidence manifest + context capsule

| # | Lane | Failure mode | Detection hint |
|---|------|--------------|----------------|
| 1 | technical | `/handback` runs in a linked story worktree, but the log is anchored to the *main* working tree via `evidence_dir()`. If `evidence-list`'s branch-slug derivation diverges from the one `evidence-append` writes under, the manifest is assembled from the wrong branch's log — or an empty one — while a populated log exists. | At audit, run `/handback` in a linked worktree on a branch with known evidence entries; the manifest must list them. Confirm `evidence-list` calls the same `evidence_dir()`/slug function `evidence-append` uses rather than re-deriving repo-root/slug locally. |
| 2 | technical | The manifest is specified to print `outputDigest`, never raw stdout/stderr — the digest exists so a failed command's output (a token, a stack trace) is never re-exposed. A record missing/empty `outputDigest` (older log, capture that failed) could trigger a raw-output fallback or a malformed row that leaks what the digest was meant to hide. | Grep the manifest-assembly code for any path that reads `predicate` output fields other than the digest. Feed a fixture record with an absent `outputDigest`; it must render a placeholder, not raw output. |
| 3 | technical | Re-running `/handback` overwrites and recommits the single per-branch file. A worker who hand-edits the Summary prose and then re-runs handback (one more test, resumed session) has the human-written narrative silently clobbered by regeneration. | Run handback, hand-edit the Summary section, re-run handback. If the edited prose is lost with no warning, the overwrite policy needs a preserve/merge rule or an explicit "regenerating, prior summary in git history" notice. |
| 4 | product | The no-log branch reports "no evidence log found." A worker on an *unarmed* branch (no work file, so `evidence-capture-hook` never fired) gets the identical message to a worker who genuinely ran nothing — the user cannot tell "capture was never on" from "you ran no commands" and may conclude their evidence was dropped. | Run `/handback` on a branch with no work file after running several verification commands. If the message cannot distinguish "hook not armed" from "no commands captured," the report conflates two states a user must tell apart. |
| 5 | product | `docs/studious/handback/<slug>.md` co-locates worker-committed, unreviewed output with review-agent output (premortems, periodic reviews) in the same `docs/studious/` tree. A human scanning that directory may read the handback as a Studious *review verdict* and over-trust an artifact the worker wrote about itself. | At acceptance, check the committed file's heading/preamble makes provenance explicit ("worker-authored evidence, not a gate verdict"). If a reader could mistake it for review output, the doc needs a provenance banner. |
| 6 | technical | The `evidence-list` read verb is claimed by this story per `evidence-format.md`'s deferral, but the sibling `gates-cite-evidence` story also reads the same log. If both land independently and each adds its own read path, the epic ends with two divergent readers of one store instead of the single location-logic owner the design promises. | At epic integration, grep for evidence-log readers. If `gates-cite-evidence` re-derives its own path instead of reusing `evidence-list`/`evidence_dir()`, the "one place each store's location logic lives" invariant broke. |
| 7 | product | `/handback` is the first command to break the naming convention's action-prefix-target shape (`gate-`, `work-on`, `extract-`). If the plan ships `/handback` while the skill shim, README, or cross-references drift toward `work-handback`, users get a split command surface for one capability. | At audit, grep the skill shim, README, and cross-references for both `/handback` and `work-handback`. A split spelling means the naming decision (flagged as an open question in the design) was never settled before build. |
